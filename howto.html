Let's begin!

Initial Setup

Saucery is built for a Continuous Integration environment. You will need to install and setup the following infrastructure as a first step:

  1. Jenkins
  2. Sauce OnDemand plugin

Contact us if you would like one of our consultants to assist you with setting up Jenkins.

Installation

  1. Create a class library project in Visual Studio. It is recommended that this be part of your application ("System Under Test") solution.
  2. Right-click and select Manage NuGet Packages for Solution...
  3. Search for Saucery2 in the Online section and click Install
  4. Add it to the class library project you created in step 1 above and click OK. Note: Only one project is to reference Saucery in your solution. Otherwise Saucery will not work correctly.

Activation

  1. Upon installation (and every time you open the solution until it is activated), the activation dialog will appear.
  2. If you have not already done so, purchase as activation key from the Full Circle website.
  3. Reenter exactly the same details into the Activation dialog.
  4. Click OK.
  5. If you entered the details correctly and validation was passed, a file called Saucery2.lic will have been written to your solution directory. Commit this file to source control as Saucery will be looking for it at runtime.

Writing your tests

Your test classes should be placed in the Tests directory and have the same namespace as GlobalSetup. Doing so will mean GlobalSetup executes once for your all your Test suites.

Let's have a look at what your test classes need to look like.

 1: using NUnit.Framework;
 2: using OpenQA.Selenium;
 3: using OpenQA.Selenium.Support.UI;
 4: using Saucery2.Tests;
 5: using Saucery2.ValueSources;
 6: using System;
 7: using System.Runtime.CompilerServices;
 8:  
 9: namespace MyApp.MyAppTests.Tests {
 10:    [TestFixture]
 11:    public class MyTests : SauceryBase {
 12:        [MethodImpl(MethodImplOptions.Synchronized)]
 13:        [Test]
 14:        public void MyFirstTest([ValueSource(typeof(PlatformTestData), "GetPlatforms")] PlatformTestData platform) {
 15:            Setup(platform);
 16:  
 17:            //
 18:            // Insert your test code here
 19:            //
 20:        }
 21:    }
 22: }

It is worth looking at this class in some more detail.

  • Line 1-7, The NUnit and Selenium related using statements will be required as you start writing your test code. The Saucery namespace using statements bring in the awesomeness.
  • Line 11: Your test class must extend SauceryBase.
  • Line 12: While not strictly required, it is recommended that you include this line for transactional reasons.
  • Line 13: This is an NUnit annotation that marks this method as a test to be executed.
  • Line 14: There's a lot going on on this line. Don't change the parameter. It handles the platform information from the Sauce OnDemand plugin.
  • Line 15: While NUnit normally would call an annotated setup method automatically, that method signature does not take any parameters, thus you need to call the Saucery Setup method with one parameter here. It ensures the DesiredCapability for the option you selected in the Jenkins job configuration is correct.
  • Line 16-onwards: Focus on your test and write the selenium code you need to robotically interact with your application.

Page Objects

At this point it is probably worth talking briefly about Page Objects. While a full discussion of the Page Object Model is outside the scope of this guide, some useful background reading is here, here and here.

Saucery provides a Page Object base class, called, as you might expect, PageObjectBase. This class gives you a head-start on writing classes that conform to the Page Object Model. Let's look at a usage example.

 1: using OpenQA.Selenium;
 2: using OpenQA.Selenium.Support.PageObjects;
 3: using OpenQA.Selenium.Support.UI;
 4: using System;
 5: using System.Globalization;
 6: using Saucery2.Driver;
 7: using Saucery2.PageObjects;
 8:  
 9: namespace MyApp.MyAppTests.PageObjects{
 10:    public class AwesomePage : PageObjectBase {
 11:        public AwesomePage(SauceryRemoteWebDriver driver, String urlRoot)
 12:          : base(driver, urlRoot + "Awesome", "Awesome", "This is the Awesome Page Title") {
 13:            GetPage(5);
 14:            CheckTitle();
 15:            PageFactory.InitElements(Driver, this);
 16:        }
 17:  
 18:        //
 19:        // Your methods that interact with elements on your page.
 20:        //
 21:  
 22:     }
 23: }

So what's happening here?

  • Line 10: You want to test your application's awesome page so you create a class called AwesomePage and extend PageObjectBase.
  • Line 11: The constructor takes two parameters: a SauceryRemoteWebDriver and the root url of the domain you are testing (e.g http://myapp.com/ or http://my.test.environment.ip.address/).
  • Line 12: The parameters are used as part of the base constructor parameters: a SauceryRemoteWebDriver, the URL to this page (the page we want to test), a name, and the expected page title.
  • Line 13: This function asks selenium to request the page and wait for it to load for the specifed number of seconds. PageObjectBase also has a default GetPage that takes no parameters and waits for 10 seconds.
  • Line 14: Asserts that the title of the page we are now on is that which we expect.
  • Line 15: This line is important and necessary if you decide to use a PageFactory class like this. There are other ways to do this which may make more sense (and result in cleaner code) for your application. More on this later.

And using your new PageObject back in your test...

 1: using MyApp.MyAppTests.PageObjects;
 2: using NUnit.Framework;
 3: using OpenQA.Selenium;
 4: using OpenQA.Selenium.Support.UI;
 5: using Saucery2.Tests;
 6: using Saucery2.ValueSources;
 7: using System;
 8: using System.Runtime.CompilerServices;
 9:  
 10: namespace MyApp.MyAppTests.Tests {
 11:     [TestFixture]
 12:     public class MyTests : SauceryBase {
 13:         [MethodImpl(MethodImplOptions.Synchronized)]
 14:         [Test]
 15:         public void MyFirstTest([ValueSource(typeof(PlatformTestData), "GetPlatforms")] PlatformTestData platform) {
 16:             Setup(platform);
 17:  
 18:             var awesomePage=new AwesomePage(Driver, Constants.TEST_URL);
 19:  
 20:             //
 21:             // Insert your test code here
 22:             //
 23:         }
 24:     }
 25: }

An added line:

  • Line 18: Call the constructor of your page object class passing in the Driver from SauceryBase and the root url of your application. As you will use it lots you might like to put it in a Constants class.

Configuring the Jenkins job

All your selected platforms should be from the WebDriver radio list. Saucery internally will direct SauceLabs to use Appium where it is the best option for the platform you selected as recommended by SauceLabs.
